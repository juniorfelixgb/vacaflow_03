## Context Summary

VacaFlow_03 is a local MVP absence management application for IGS Solutions. The entire application must run from source code on a reviewer's developer machine with no cloud infrastructure, containers, or external services. The mandated architecture style is a **Reduced Onion Architecture** with exactly five layers (Domain, Application, Infrastructure, API, Web), using Next.js for the web layer and ASP.NET Core Minimal API for the backend, with SQLite as the persistence engine.

The three alternatives below differ not in the mandated five-layer structure — which is a hard technical constraint per NFR-MAINT-001 — but in **how authentication state is managed and propagated**, **how the API layer communicates identity to the Application layer**, and **how the Infrastructure layer surfaces repositories**. These are the genuine architectural decision points remaining within the constrained scope of the VacaFlow MVP.

**Top Quality Attributes driving selection (from UT-001):**
1. Security (Critical / Highest Weight) — server-side identity authority, password hashing, role enforcement
2. Reliability (Critical / High Weight) — transactional approval records, consistent state machine enforcement
3. Usability & Compatibility (High / Joint Third) — clean HTTP boundary, local self-contained execution
4. Maintainability (Medium) — no MediatR, CQRS, generic repos; five named layers identifiable in code

## Alternative A: Cookie-Based Session Authentication with Concrete Service Injection

### A.1 Concept

This alternative implements authentication using ASP.NET Core's built-in cookie authentication middleware, storing the authenticated user's identity in an encrypted HTTP-only cookie issued at login. The Application layer receives the current user's identity via a `SessionContext` service that wraps `IHttpContextAccessor`, injected as a scoped dependency into all Application services. Repository interfaces are declared in the Application layer and implemented with concrete EF Core classes in the Infrastructure layer, registered directly by type in the DI container with no abstraction over the unit of work.

This is the most conventional and framework-aligned approach available within the Onion constraints, minimising the number of custom abstractions while fully satisfying all security, identity, and reliability requirements defined in NFR-001.

### A.4 Deployment Model

**Target:** Single developer machine running Windows, macOS, or Linux.

**Execution model:** Two processes started from the terminal — `dotnet run` for the ASP.NET Core API (default port 5000 or 7000 with HTTPS) and `npm run dev` for the Next.js web application (default port 3000). No containers, reverse proxies, or process managers are required.

**Data storage:** A single `vacaflow.db` SQLite file written to the API project root or a configured path. EF Core migrations or `EnsureCreated` run automatically at startup. The file is listed in `.gitignore` and is never committed to source control.

**Authentication state:** The ASP.NET Core cookie is issued at login with `HttpOnly`, `SameSite=Strict`, and a configurable expiration (e.g. sliding 60-minute session). The Next.js app includes `credentials: 'include'` on every `fetch` call to ensure the cookie travels with cross-origin requests (CORS configured to allow the Next.js origin with `AllowCredentials`).

**Environment:** Single environment (`Development`). No staging or production environment is provisioned within this MVP.

### A.5 Strengths

- **Highest alignment with NFR-SEC-002 and NFR-SEC-003**: The `SessionContext` pattern guarantees that employee and approver identity is always read from the server-side cookie claim, never from the request body. This is the most direct implementation of the "Server-Side Identity Authority" architecture principle.
- **Zero custom authentication infrastructure**: ASP.NET Core's built-in cookie authentication handles encryption, tamper detection, and expiration natively, reducing the risk of a custom implementation introducing vulnerabilities.
- **Transactional reliability out of the box**: EF Core's `SaveChangesAsync` within a repository method naturally wraps the state update and `ApprovalRecord` insert in a single SQLite transaction, satisfying NFR-REL-002 without requiring explicit `BeginTransaction` calls.
- **Minimal abstraction overhead**: Concrete repositories registered directly in DI — no `IUnitOfWork` or generic `IRepository<T>` — complies with NFR-MAINT-001 and keeps the codebase readable for a reviewer unfamiliar with the project.
- **Straightforward local setup**: Cookie-based sessions work out of the box with browser `fetch` and CORS `AllowCredentials`, requiring no token storage logic on the Next.js side.

### A.6 Weaknesses

- **CORS configuration is required for cookie credentials**: Cross-origin `credentials: 'include'` requires explicit CORS policy with `AllowCredentials` and a named origin allowlist; misconfiguration causes silent authentication failures in the browser, which can be hard to debug.
- **Cookie expiration must be tuned**: Default ASP.NET Core cookie settings may be overly long for a review environment or may expire mid-session during a reviewer's walkthrough if set too short; requires deliberate README documentation.
- **Not portable beyond the local machine**: The cookie domain and SameSite settings make this authentication model unsuitable for multi-host or cloud deployment, limiting future migration paths.
- **`IHttpContextAccessor` leaks HTTP context into the Application layer**: While pragmatic and widely used, this couples the Application layer to the HTTP request lifecycle, which is a purity concern if the architecture ever needs to support background tasks or CLI operations.
- **No built-in token introspection endpoint**: Unlike JWT, cookie-based sessions do not produce a self-contained token that can be inspected without a running server, making it harder to debug identity issues in isolation.

## Alternative B: JWT Bearer Token Authentication with ICurrentUserService Interface

### B.1 Concept

This alternative replaces the cookie-based session with JWT (JSON Web Tokens) issued at login by the ASP.NET Core API. The token is stored in memory (or `sessionStorage`) on the Next.js side and sent as a `Bearer` header on every API request. Rather than using `IHttpContextAccessor` directly in the Application layer, the identity is surfaced through an `ICurrentUserService` interface declared in the Application layer and implemented in the API layer — keeping the Application layer free of any HTTP-specific dependency while still satisfying the server-side identity authority requirement.

This alternative introduces a slightly higher degree of layering discipline compared to Alternative A, at the cost of requiring token storage and refresh management on the frontend.

### B.4 Deployment Model

**Target:** Single developer machine (Windows, macOS, or Linux).

**Execution model:** Same two-process model as Alternative A — `dotnet run` for the API and `npm run dev` for Next.js. No containers or infrastructure required.

**Authentication state:** JWT tokens are stateless; the server holds no session table. The token is issued at login, stored on the client in React context (lost on tab close) and optionally persisted to `sessionStorage` (cleared on browser session end). The token lifetime is configured in `appsettings.Development.json` (recommended: 60 minutes). Logout is handled by the client discarding the token; no server-side revocation is implemented in the MVP.

**JWT signing key:** A symmetric key (HS256) is stored in `appsettings.Development.json` under a key not committed to source control (`.gitignore` entry for the file or use of `dotnet user-secrets`). The README documents how to generate and configure the key.

**CORS:** Because the Next.js app sends an `Authorization` header rather than a cookie, CORS configuration is simpler — `AllowAnyHeader` can be set without `AllowCredentials`, though the origin allowlist should still be restricted to `http://localhost:3000`.

**Data storage:** Same SQLite file strategy as Alternative A.

### B.5 Strengths

- **Cleaner Application layer boundary**: `ICurrentUserService` is a pure interface with no HTTP-specific types; Application services can be tested without mocking `IHttpContextAccessor` or instantiating an `HttpContext`, directly supporting NFR-REL-003 test coverage.
- **No CORS `AllowCredentials` required**: JWT Bearer authentication does not rely on cookies, eliminating the CORS credential complexity that can cause subtle authentication failures in Alternative A.
- **Stateless server — simpler horizontal scaling if needed post-MVP**: No session store or affinity requirement; while scaling is out of scope for the MVP, this decision leaves the door open without architectural debt.
- **Stronger separation of concerns at the Application layer**: The `ICurrentUserService` abstraction means the Application layer does not depend on ASP.NET Core types, satisfying the Onion dependency rule more strictly than the `IHttpContextAccessor` injection in Alternative A.
- **No EF Core Identity extension tables required**: Authentication uses only the `Employees` table with a hashed password column, reducing schema complexity and eliminating the additional EF Core Identity migration surface.

### B.6 Weaknesses

- **Token storage on the client is a security consideration**: Storing the JWT in `sessionStorage` is safer than `localStorage` but still exposes it to XSS attacks; cookie-based tokens (with `HttpOnly`) are immune to this class of attack. For an MVP reviewer environment this risk is acceptable, but it must be acknowledged in the README.
- **No server-side logout revocation**: Once a JWT is issued, it remains valid until expiry regardless of logout. For a review-window MVP this is acceptable, but it means a stolen token cannot be invalidated without restarting the server.
- **Additional setup step for JWT signing key**: The reviewer must configure or generate a signing key before the application starts, adding one step to the README setup sequence and a potential source of startup failure.
- **`JwtTokenGenerator` is custom code**: Implementing correct JWT signing and validation requires care; a misconfigured token lifetime or missing audience/issuer validation weakens the security posture compared to the fully managed cookie middleware in Alternative A.
- **React context token management adds frontend complexity**: Hydrating the user context from `sessionStorage` on page load, handling 401 redirects, and managing token expiry in the UI requires more frontend scaffolding than reading a session cookie.

## Alternative C: Cookie-Based Authentication with Explicit Transaction Service and ICurrentUserContext

### C.1 Concept

This alternative combines cookie-based authentication (as in Alternative A) with an explicit `ICurrentUserContext` interface declared in the Application layer (as in Alternative B) AND an explicit `ITransactionService` interface that makes transactional boundaries a first-class Application-layer concern rather than relying on EF Core's implicit `SaveChangesAsync` behavior. The result is the highest-fidelity implementation of the Onion dependency rule across all five layers: the Application layer depends only on interfaces with no HTTP or EF Core types anywhere in its assembly, while the Infrastructure layer provides all concrete implementations.

This alternative is the most architecturally pure of the three — it is the most testable, the most explicit about transaction semantics, and the cleanest expression of the five-layer Onion model — at the cost of slightly more interface declarations and DI registrations.

### C.4 Deployment Model

**Target:** Single developer machine (Windows, macOS, or Linux).

**Execution model:** Same two-process model as Alternatives A and B — `dotnet run` for the API and `npm run dev` for Next.js. No additional infrastructure.

**Authentication state:** Same cookie-based model as Alternative A. `HttpOnly`, `SameSite=Strict`, sliding 60-minute expiration, `credentials: 'include'` on the Next.js side.

**Transaction semantics:** `EfCoreTransactionService` uses SQLite's default serializable isolation for the explicit transaction wrapping approve/reject. Because the MVP assumes no concurrent load (per NFR-PERF-002 and the single-reviewer assumption), contention is not a concern.

**Data storage:** Same SQLite file strategy as Alternatives A and B.

**CORS:** Same configuration as Alternative A — `AllowCredentials` with a restricted origin.

### C.5 Strengths

- **Strictest Onion boundary compliance**: The Application layer assembly has zero dependencies on ASP.NET Core, EF Core, or any infrastructure types. This is the most faithful implementation of the five-layer Onion model defined in NFR-MAINT-001.
- **Most testable Application layer**: Because both `ICurrentUserContext` and `ITransactionService` are pure interfaces, all Application services can be unit-tested with simple in-memory fakes — no `HttpContext`, no `DbContext`, no EF Core test doubles needed.
- **Explicit, visible transaction boundary**: `ITransactionService.ExecuteInTransactionAsync` makes the atomic unit explicit in the Application service code, satisfying NFR-REL-002 in a way that is immediately understandable to a reviewer reading `ApprovalService` without knowledge of EF Core internals.
- **Infrastructure-agnostic Application layer**: If SQLite were replaced with a different database post-MVP, only the `EfCoreTransactionService` and concrete repositories would change; no Application service code would need to be updated.
- **Combines best-practice elements from both A and B**: Cookie authentication (simple client setup, `HttpOnly` security) with the `ICurrentUserContext` abstraction (Application layer independence from HTTP types).

### C.6 Weaknesses

- **More interfaces and DI registrations than the other alternatives**: Six interfaces are declared in the Application layer (`ICurrentUserContext`, `ITransactionService`, plus four repositories), each requiring a concrete implementation and a DI registration. This is the most verbose setup of the three alternatives.
- **`ITransactionService` abstraction may be over-engineering for a single-database MVP**: EF Core's `SaveChangesAsync` is already transactional for a single SQLite context; wrapping it in an explicit interface adds ceremony that a reviewer unfamiliar with the pattern may find confusing.
- **Same CORS cookie credential complexity as Alternative A**: `credentials: 'include'` and `AllowCredentials` CORS configuration are required, carrying the same misconfiguration risk described in Alternative A.
- **Slightly longer initial development time**: The additional interface layer means more classes to create and wire up compared to Alternative A, which could be a factor if the reviewer is also the implementer.
- **SQLite's limited concurrent transaction support still applies**: The explicit transaction in `EfCoreTransactionService` provides correctness guarantees but does not change SQLite's serialized write model; under any real concurrent load the explicit transaction would contend just as the implicit one would.
