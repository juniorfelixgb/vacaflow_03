## 6. Recommendation

### Primary Recommendation: Alternative C — Cookie-Based Authentication with Explicit Transaction Service and ICurrentUserContext

Alternative C is the optimal architecture for VacaFlow_03, achieving the maximum weighted score of 5.000 across all four quality attributes. It delivers the strictest compliance with the Reduced Onion Architecture mandate (NFR-MAINT-001) by ensuring that the Application layer assembly carries zero dependencies on ASP.NET Core or Entity Framework Core types — a property that is both architecturally correct and directly verifiable by the reviewer by inspecting the `using` directives of any Application layer class. At the same time, it inherits the full Security profile of Alternative A (cookie `HttpOnly`, no token-theft surface) and the same frictionless local setup (two commands, no external secrets to configure). The explicit `ITransactionService` makes the atomic approval boundary visible in the `ApprovalService` code itself rather than relying on EF Core's implicit commit behavior, directly satisfying NFR-REL-002 in a way that is immediately understandable to any reviewer regardless of their EF Core experience.

**Why Alternative C wins:**

- **Security (5.00):** Cookie `HttpOnly` with `SameSite=Strict` prevents token exfiltration via XSS across all four security scenarios. The `ICurrentUserContext` interface read exclusively from the server-validated cookie claim ensures that approver identity can never be injected from the request body (SC-SEC-04). Role enforcement via `RoleAuthorizationMiddleware` and domain-level `SelfApprovalGuard` provide defense in depth. Alternative C matches Alternative A's perfect security score.

- **Reliability (5.00):** The explicit `ITransactionService.ExecuteInTransactionAsync` wrapping the approve/reject operation in a `BeginTransactionAsync` / commit / rollback block is the decisive differentiator. SC-REL-01 (atomic approval write) receives a score of 5 because the transactional guarantee is structurally enforced and visibly documented in the Application service code, not inferred from EF Core behavior. Auto-provisioning via EF Core seed data on first run (SC-REL-02) and the Domain `RequestStateMachine` pre-persistence guard (SC-REL-03) operate identically to Alternative A.

- **Usability & Compatibility (5.00):** Identical to Alternative A — two-process startup with no external configuration, cross-platform runtime stack, and state-driven UI action visibility. Alternative C adds no setup steps over Alternative A.

- **Maintainability (5.00):** The Application layer's zero-dependency profile makes all five Onion layer boundaries immediately identifiable from `using` directives (SC-MAINT-01). Both `ICurrentUserContext` and `ITransactionService` are pure interfaces, enabling full Application layer unit test coverage with simple in-memory fakes and no `HttpContext` or `DbContext` involvement (SC-MAINT-02). Repository swaps, user context swaps, and transaction implementation swaps each require a single DI registration change in the API layer (SC-MAINT-03).

**Why not Alternative A: Cookie + SessionContext:**

Alternative A is a strong, production-grade implementation and a legitimate choice. Its Achilles' heel for VacaFlow_03 is the Maintainability dimension: `SessionContext` injects `IHttpContextAccessor` — an ASP.NET Core infrastructure type — into the Application layer. This violates the strict Onion dependency rule that the Application layer must have no outward dependency on framework types. For a project whose primary purpose is to demonstrate architectural fitness to a reviewer, this is a meaningful trade-off: a reviewer reading the Application layer will see `Microsoft.AspNetCore.Http` in the `using` directives and correctly note the boundary violation. Alternative A would be the right choice if the team prioritizes convention-first simplicity and expects no unit testing of Application services during the MVP review window.

**Why not Alternative B: JWT + ICurrentUserService:**

Alternative B introduces the cleanest Application layer separation among the three alternatives that use only a single new interface (the `ICurrentUserService` abstraction without `ITransactionService`), and it is the best choice if the project were to evolve into a multi-host or stateless deployment post-MVP. However, for the constrained VacaFlow_03 context, it carries two meaningful costs: first, the JWT stored in `sessionStorage` is accessible to JavaScript, creating an XSS token-theft surface that `HttpOnly` cookies eliminate — this is not a theoretical concern for a developer-machine MVP, but it is the correct architectural observation; second, the JWT signing key must be configured before the API will accept any login, adding a friction point to the reviewer setup flow that directly conflicts with SC-USE-01's 15-minute target. Alternative B would be the right choice if the post-MVP roadmap included horizontal scaling, multi-service token validation, or the team had a strong preference for stateless server design.

### Strategic Path Forward

1. **Phase 1 — Local MVP Delivery:** Implement Alternative C as specified. Configure `EnsureCreated` or an initial EF Core migration for SQLite auto-provisioning, wire `EfCoreTransactionService` and `HttpContextCurrentUserContext` in the API layer DI, and verify all domain rules through Application layer unit tests using in-memory fakes. Deliver the working application for reviewer evaluation.

2. **Phase 2 — Architecture Hardening (Post-Review):** Replace `EfCoreTransactionService` with a resilience-enhanced implementation using EF Core's `CreateExecutionStrategy` for retry-aware transaction semantics. Add integration tests that exercise the full request lifecycle (register → create → submit → approve) against an in-memory or file-based SQLite database to validate the transaction boundary under simulated fault conditions.

3. **Phase 3 — Production Migration (If Required):** If VacaFlow_03 evolves beyond a single-machine MVP into a hosted service, replace `HttpContextCurrentUserContext` with an equivalent implementation backed by a distributed identity provider (e.g., Azure AD / Entra ID), replace `EfCoreTransactionService` with a PostgreSQL or SQL Server backed implementation, and replace the SQLite `VacaFlowDbContext` with a production-grade provider — all without changing a single line of Application layer code, validating the investment in Onion boundary purity made in Phase 1.

## 7. Risks & Mitigations

| Risk | Probability | Impact | Mitigation Strategy |
|------|-------------|--------|---------------------|
| **CORS `AllowCredentials` misconfiguration causes silent auth failures:** Cookie `credentials: 'include'` on Next.js requires the API's CORS policy to explicitly name the allowed origin (`http://localhost:3000`) and set `AllowCredentials`; if the origin is wrong or `AllowCredentials` is omitted, the browser silently strips the cookie and all API calls return 401 with no clear error. | Medium | High | Document the exact CORS configuration in the README with a copy-paste snippet; add a startup health-check endpoint (`GET /health`) that the README instructs the reviewer to call from the browser console to verify cookie round-trip before the first login attempt. |
| **Cookie sliding expiration interrupts a reviewer's walkthrough:** If the reviewer pauses a demo session longer than the configured cookie lifetime, subsequent API calls silently return 401; the reviewer may interpret this as an application bug rather than a session expiry. | Medium | Medium | Set a generous sliding expiration (120 minutes) in `appsettings.Development.json` and document the setting prominently in the README. Log a startup message displaying the configured expiration so the reviewer is aware of the value. |
| **`EfCoreTransactionService` wraps an already-opened EF Core context:** EF Core SQLite provider does not support nested transactions; if `ExecuteInTransactionAsync` is called while a `DbContext` already has an open transaction from a parent scope, it will throw `InvalidOperationException`. | Low | High | Document that `ITransactionService` is for top-level Application service operations only; add a guard in `EfCoreTransactionService` that throws a descriptive `InvalidOperationException` with a message explaining the nesting restriction if a transaction is already active on the context. |
| **`ICurrentUserContext` returns null UserId before cookie is populated:** If a code path calls `ICurrentUserContext.CurrentUserId` before the cookie middleware has run (e.g., on an accidentally unprotected endpoint or during startup), the returned GUID will be `Guid.Empty`, potentially allowing unauthorized operations to proceed silently. | Low | High | Add a guard in `HttpContextCurrentUserContext.CurrentUserId` that throws `UnauthorizedAccessException` if the claim is absent, rather than returning `Guid.Empty`; write a unit test for the guard in the Application layer using a fake that simulates the absent-claim scenario. |
| **SQLite `.db` file committed to source control exposes seed passwords:** The `ManagerAccountSeeder` writes a BCrypt-hashed password to the database; if `vacaflow.db` is committed, the hash is publicly visible and could be brute-forced if a weak password is used. | Low | Medium | Enforce `.gitignore` entry for `*.db` and `*.db-shm` and `*.db-wal` in the API project directory; add a `dotnet run` startup check that logs a warning if the database file is detected outside the expected path; document the `.gitignore` rule in the README. |
| **`ITransactionService` abstraction confuses reviewers unfamiliar with the pattern:** A reviewer not familiar with the Onion transaction service pattern may question why the Application layer declares an interface for something EF Core handles implicitly, potentially interpreting the design as over-engineered for an MVP. | Medium | Low | Add a one-paragraph comment in `ITransactionService.cs` and in `ApprovalService.cs` explaining that the interface makes the atomic boundary explicit and removes the Application layer's dependency on EF Core's implicit commit behavior, citing NFR-REL-002 and NFR-MAINT-001 as drivers. |

## 9. Future Considerations

1. **Migration to a Hosted Database (PostgreSQL or SQL Server)**
   VacaFlow_03 uses SQLite exclusively because the MVP runs on a single developer machine with no concurrent write load. If the application is deployed to a shared environment or needs to support multiple simultaneous users, SQLite's serialized write model becomes a bottleneck. The `ITransactionService` and `IRepository` abstractions in Alternative C make this migration surgical: only `EfCoreTransactionService` and the four concrete repository classes need to change, while all Application layer services remain untouched.
   *Trigger:* User load exceeds 5 concurrent sessions, or the application is deployed to a server accessible by more than one user at a time.

2. **Token-Based Authentication for API Consumers**
   Cookie-based authentication is optimal for browser-to-API communication, but if VacaFlow_03 evolves to expose its API to mobile clients, CI/CD pipelines, or third-party integrations, cookie credentials become impractical. The `ICurrentUserContext` interface in Alternative C provides a clean extension point: a JWT-backed `JwtCurrentUserContext` can be registered alongside the cookie-backed implementation for specific endpoint groups, or the cookie implementation can be fully replaced without touching Application layer code.
   *Trigger:* A non-browser client (mobile app, external integration, or automation script) needs to call the VacaFlow API.

3. **Introduction of an Outbox Pattern for Notification Delivery**
   As VacaFlow_03 matures, stakeholders will likely require email or in-app notifications on request state changes (e.g., "Your request has been approved"). Integrating notification dispatch directly into `ApprovalService` risks coupling the core approval transaction to an external service call that could fail or slow the response. An outbox pattern — where a `NotificationOutboxRecord` is written atomically inside the same `ITransactionService` transaction and a background worker dispatches it — decouples delivery from the approval flow. The explicit `ITransactionService` in Alternative C makes it straightforward to extend the transactional unit with the outbox insert.
   *Trigger:* A stakeholder requests email or in-app notifications for any request lifecycle event.

4. **Soft-Delete and Audit Trail for Domain Entities**
   The current implementation uses hard deletes where applicable and does not maintain a history of entity state changes beyond `ApprovalRecord`. For compliance or HR audit purposes, the `AbsenceRequest` table may need to retain all historical states, and the `Employee` table may need soft-delete semantics (deactivation rather than physical deletion). Because all data access flows through the Application layer interfaces, adding a global EF Core query filter for soft-deleted records and an `AuditLog` entity requires changes only in `VacaFlowDbContext` and the concrete repository implementations — no Application or Domain changes.
   *Trigger:* An HR compliance requirement is identified that mandates retention of historical request data or employee deactivation audit records.

5. **Containerisation for Reproducible Reviewer Environments**
   The VacaFlow_03 MVP explicitly excludes Docker and containers to minimise reviewer setup friction. However, if the project is shared with a larger reviewer group or integrated into a CI/CD pipeline for automated acceptance testing, containerisation becomes valuable. Because Alternative C uses no host-specific paths (SQLite path via `Path.Combine`) and all configuration is environment-variable compatible, a `Dockerfile` for the API and a `docker-compose.yml` can be added without architectural changes; the `ITransactionService` and `ICurrentUserContext` bindings in `Program.cs` require no modification.
   *Trigger:* The project is shared beyond a single reviewer machine, or an automated CI pipeline running acceptance tests is introduced.

6. **Role Hierarchy Expansion Beyond Employee / Manager**
   VacaFlow_03 defines exactly two roles: `Employee` and `Manager`. If the business introduces additional roles (e.g., `HRAdmin`, `Director`, `TeamLead` with delegated approval authority), the current `UserRole` value object and `RoleValidator` domain rule need to accommodate additional states. The `RoleAuthorizationMiddleware` policies would need to be updated, and `ICurrentUserContext.CurrentUserRole` would need to return an enumeration or string that the domain can evaluate. The Application and Domain layers are insulated from the HTTP representation of roles, so this expansion touches only `UserRole`, `RoleValidator`, and the middleware authorization policies.
   *Trigger:* A business requirement introduces a third user role with distinct permissions or approval delegation logic.
