03 Practical example & getting started

One real requirement, start to finish

StageSourceContent
RequirementFR-AA-001"The system shall authenticate users via a username and password login form." (Must Have, FRS-001 §3.9)
Test caseUS-001 / AC-001Given a registered, active user enters a valid username and correct password, When they submit the login form, Then the system validates the hash, issues a JWT with role claims, and redirects to the dashboard.

Illustrative code (AAA template from spec-authoring-aaa.md)

Nothing has been generated yet for Contoso — this instantiates the toolkit's real spec template with the AC above.

import { test, expect } from '@playwright/test';
import { LoginPage } from '../../pageObjects/LoginPage';
test.describe('Authentication', () => {
  test('TC-001: valid credentials redirect to dashboard', async ({ page }) => {
    const loginPage = new LoginPage(page);
    await loginPage.goto();
    await loginPage.login(process.env.USERNAME!, process.env.PASSWORD!);
    await expect(page).toHaveURL(/\/dashboard/);
  });
});

Checklist to start

Review the 3 real SDLC docs

project/01-understand/strategic-intake.md, project/02-define/Functional_Requirements_Specification_V1_0.md, project/05-planning/Backlog.md.

Run /qa-docs → DISCOVER

Inventories those 3 artifacts against the intake checklist.

Continue → DECISIONS

Approve the scored strategy option at the APPROVAL GATE (qa-decisions.md).

Continue → GENERATE

Approve each of the 5 gated documents, CP-STRATEGY through CP-REPORT.

Run /qa-setup

Turns the approved strategy into docs/qa-config.yaml, approved at CP-1.

Run /qa-orchestrator

Routes the first real item (e.g. FR-AA-001) into a /qa-lane.
