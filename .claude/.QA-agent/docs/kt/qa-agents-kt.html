01 Why this exists

The cost of writing tests collapsed. The job changed.

Traditional QA was designed for a world where automating a test was slow and expensive. AI changed that. The bottleneck is no longer typing the test — it's deciding what to test, validating it with human judgment, and closing the loop.

Traditional QA

  • Write each test by hand, line by line
  • Automation is a separate, scarce specialty
  • Coverage tracked in spreadsheets
  • Flaky tests fixed reactively, one by one
  • QA enters late, at the "testing phase"

AI-native QA (what we have)

  • Agents draft tests; you direct and approve
  • Every QA can drive automation through plain requests
  • Coverage enforced automatically (gate stops at gaps)
  • Failures diagnosed and healed by a dedicated agent
  • QA shifts left (early) and right (production feedback)
The one idea to keep

Your testing judgment is the steering wheel; the agents are the engine. You spend less time on boilerplate and more on the high-value work only a tester can do — deciding what to test, and judging whether the result is truly right.

02 Mental model

What is an "agent", really?

It all runs inside Claude Code (the AI assistant in our editor). The toolkit gives Claude three kinds of building blocks. One analogy: cooking.

📋

Command

The recipe

A slash command like /qa-lane regression. It's an ordered procedure that drives a whole job from start to finish and pulls in the right skills at each step.

🧩

Skill

The technique

A reusable capability like writing a feature file or choosing a good locator. Commands (and Claude) call skills; skills know the rules.

🔬

Subagent

The specialist you call in

A focused worker (review, healing) that runs in its own room so its noisy details don't clutter your chat — and reports back a clean verdict.

Two things that make it trustworthy

Config-driven: paths, commands and thresholds live in one file (docs/qa-config.yaml) — nothing is hardcoded. Human-gated: the agents stop at checkpoints and wait for your Approve / Edit / Reject before doing anything that matters.

04 How it's built

Three layers, and one golden rule

You don't need to memorize this — but knowing the shape explains why the agents stay consistent and easy to change.

LayerRoleAnalogyExample
CommandsThin orchestrators — route work, enforce gatesThe recipe/qa-lane acceptance
SkillsThin wrappers that apply the rules to do a taskThe techniquegherkin-authoring
Reference docsCanonical rules — the only place a rule is writtenThe cookbookgherkin-standards.md
Golden rule — "rules live once"

If a standard changes (say, a new locator strategy), it is updated in .claude/docs/reference/ only — never copied into a command or a skill. That is why the whole toolkit stays in sync and never contradicts itself.

And never hardcode

Paths, Playwright project names, run commands, env-var lists and thresholds all come from docs/qa-config.yaml. Move the toolkit to a new project, edit one file, and it works.

05 The three lanes

08 The subagents

Two specialists that work in their own room

Some jobs are noisy — lots of stack traces and retries. Subagents run them in an isolated context and hand back a short, clean report, so your chat stays readable.

🔍 qa-reviewer · gate CP-3

Quality review before merge

Runs the three-level review — static analysis, mutation testing, execution validation — and returns a verdict with findings classified BLOCKER / CRITICAL / MAJOR / MINOR. Reports only; never edits code. A BLOCKER fails the wave.

🩹 qa-healer · gate CP-4

Diagnose & fix failing tests

Classifies each failure (LOC / STR / TIME / MISS / MTH / IMP), applies the minimum fix, re-runs, up to 3 attempts per test. Unresolved tests escalate — a real app bug becomes a bug report, not a weakened assertion.

The gates that protect quality

100% AC coverage — the run hard-stops if any acceptance criterion would be left uncovered. Max 10 test cases per wave — bigger work is split. PR checkpoint between waves when enabled.

11 The autonomous pipeline

/qa-loop — generate → implement → heal

When you want the whole job done for one work item, /qa-loop runs three phases on its own and stops for your review between phases (never mid-phase).

PHASE 1
Test-case generation
max 5 iterations

From an ADO item, a PBI, or an existing feature: build the coverage list and close gaps to 100%.

↓ Gate: Approve / Reject / Stop
PHASE 2
Implementation
max 3 iterations / wave

Reuse & extend page objects and steps, implement, run, apply minimum healing, then qa-reviewer checks the wave.

↓ Gate: Approve / Reject / Skip / Stop
PHASE 3
Healing
max 3 attempts / test

qa-healer takes the remaining failures, fixes per the taxonomy, and escalates anything unresolved.

↓ Gate: Approve / Reject / Stop

Each phase writes a log (phase1-generation.md, phase2-implementation.md, phase3-healing.md) and the run ends with a final summary.

15 Glossary & cheat sheet

The words you'll hear

POM
Page Object Model — UI screens wrapped as reusable classes so tests read clean and locators live in one place.
BDD / Gherkin
Behaviour-Driven Development. Scenarios written in plain Given/When/Then language in .feature files.
AAA
Arrange-Act-Assert — the three-part shape of a clean test: set up, do the action, check the result.
Locator priority
getByRole → getByLabel → getByTestId → getByText → CSS. Never XPath, nth() or first().
Lane
A track of automation with its own folder, command and output. Its name is a free label — its style (pom-spec / bdd-gherkin / api-request) decides the pipeline.
Wave
One batch of up to 10 test cases, kept small so each review fits a human screen.
Coverage matrix
The AC → test-case mapping. The gate that blocks if any criterion is uncovered.
Mutation testing
Deliberately breaking the code to confirm your tests would catch it — a quality check on the tests themselves.
Healing taxonomy
LOC (locator), STR (missing step), TIME (timeout), MISS (missing data), MTH (missing method), IMP (test-code bug).
CP-NN
A checkpoint — the agent pauses for your Approve / Edit / Reject before continuing.
ADO
Azure DevOps — where work items, acceptance criteria and bugs live.
Shift-left / shift-right
Test earlier (requirements/design) and learn later (production monitoring & feedback).
